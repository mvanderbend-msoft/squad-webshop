# Squad Demo — Orchestration Guide

This guide drives the live "Squad Demo" using this webshop repo as the starting point. The demo uses [**Squad**](https://github.com/bradygaster/squad) — a real human-led AI agent team framework that runs through GitHub Copilot.

The narrative arc:

> *"You're in a git repository. Push it to GitHub, set up a Squad team, write me a PRD as a single GitHub issue, split it into work items, and Ralph — please start working. Don't commit to main directly, always open a PR."*

The feature being built live is **Add to Favorites**. The repo deliberately does **not** include it.

---

## 1. How Squad actually works (60-second mental model)

| Concept | What it is | Where it lives |
|---|---|---|
| **The team** | Persistent, *named* agents (e.g. `edie`, `mcmanus`, `keaton`) — each with a charter and history. Generated by `squad init` from a casting policy. | `.squad/agents/{name}/charter.md` |
| **Coordinator (`Squad`)** | Routes your prompts to the right member(s). Spawns multiple in parallel when relevant. | `copilot --agent squad` |
| **Lead** | The triager. Reads a `squad`-labeled issue, decides who owns it, and replaces the label with `squad:{member}`. | A specific named team member |
| **Ralph** | Polling loop. Watches GitHub for `squad:{member}` issues, dispatches the matching specialist agent to work them, monitors PRs. | `squad triage --execute` |
| **Specialists** | The actual workers. They read their charter, work in their voice, follow `squad/{issue}-{slug}` branch + PR convention. | Spawned by Ralph (or the coordinator) |
| **Scribe** | Silent memory manager — logs decisions and agent learnings back to `.squad/`. | Always running |

Key labels Squad uses on GitHub issues (already in `.github/labels.yml`):

- `squad` — the inbox; Lead triages anything wearing it
- `squad:{member}` — assigns to a specific named member (the label is *created* at `squad init` time once the cast is named — you don't pre-make them)
- `squad:untriaged`, `priority:p0–p3`, `next-up`, `go:needs-research`

## 2. Pre-demo checklist

Run through this **before** going on stage.

- [ ] Node.js 22+ and npm installed (`node -v`)
- [ ] `gh` CLI authenticated against the demo account: `gh auth status`
- [ ] GitHub Copilot CLI installed: `copilot --version`
- [ ] Squad CLI installed globally:
  ```bash
  npm install -g @bradygaster/squad-cli
  squad --version
  ```
- [ ] Repo cloned and the app actually runs:
  ```bash
  cd squad-webshop
  npm install
  npm run dev
  # http://localhost:5173 — register, browse, checkout. Confirm it works.
  ```
- [ ] No GitHub repo with the same name yet (Step 3 below creates it). If you've run the demo before, see **§7 Reset**.
- [ ] `git status` is clean.

### Optional, recommended polish
- **Branch protection on `main`** so direct pushes fail. Easiest via the UI: Settings → Branches → Add rule for `main` → "Require a pull request before merging". This makes the "always open a PR" rule a hard guarantee rather than a social one.
- Big terminal font, clean shell, browser tab open to your GitHub profile.

## 4. The demo script

Each section below is a discrete on-stage moment. Copy the prompts verbatim.

### Step 1 — Push to GitHub

Outside any agent, in a regular terminal:

> **Spoken:** *"We're in a git repo on my laptop. Let's get it on GitHub."*

```bash
gh repo create squad-webshop --public --source=. --remote=origin --push
```

(Or, if you want Copilot CLI to do it for the cinematic effect:
> *"You're in a git repository. Create a public GitHub repository called `squad-webshop` for it and push everything."*)

Then sync labels (off-camera ideally — it isn't interesting):

```bash
gh label create "squad"             -c "5319e7" -d "Squad backlog — Lead will triage" --force
gh label create "squad:untriaged"   -c "fbca04" -d "Needs triage by the Lead" --force
gh label create "priority:p0"       -c "b60205" -d "Critical" --force
gh label create "priority:p1"       -c "d93f0b" -d "High" --force
gh label create "priority:p2"       -c "fbca04" -d "Medium" --force
gh label create "priority:p3"       -c "0e8a16" -d "Low" --force
gh label create "next-up"           -c "0e8a16" -d "Queued for next agent pickup" --force
gh label create "go:needs-research" -c "1f6feb" -d "Needs research first" --force
```

### Step 2 — Hire the team

> **Spoken:** *"Now let's bring in the team."*

```bash
squad init
gh auth status     # ✓ logged in
squad copilot      # add @copilot as the coding agent (optional but recommended)
```

`squad init` scaffolds `.squad/` with `team.md`, `routing.md`, `decisions.md`, an `identity/`, a `casting/` policy, and empty `agents/{name}/` charters with thematically named members.

### Step 3 — Open the team's chat

```bash
copilot --agent squad --yolo
```

> **Why `--yolo`?** Squad makes a *lot* of tool calls per session. Without it Copilot prompts to approve each one and your demo dies of paper cuts.

You'll get a Copilot prompt with the Squad agent loaded. Greet the team:

> **Prompt:**
> *"I'm starting work on this webshop. Set up the team for a React + Node/Express + SQLite project. The next feature we're building is 'Add to Favorites'."*

Squad proposes the cast — say **yes**. Now you have named specialists with charters committed to git.

### Step 4 — Have the team write the PRD as a single GitHub issue

Still inside `copilot --agent squad`:

> **Prompt:**
> *"Lead — open a single GitHub issue that is the PRD for an 'Add to Favorites' feature. Use the PRD issue template. Users should be able to favorite products from anywhere they appear and have a dedicated `/favorites` page. Cover database, backend API, and frontend. Apply the `squad` label so it shows up in the inbox."*

Expected: one issue opens with PRD sections (summary, user stories, scope, AC, cross-cutting work, notes). Single issue, `squad` label.

### Step 5 — Split the PRD into labeled work items

> **Prompt:**
> *"Lead — split that PRD into individual work items, one issue per area (database, backend, frontend, tests, docs). Each child issue should reference the parent PRD, use the work-item template, start with the `squad` label, and then you triage each one and replace `squad` with the right `squad:{member}` label so the named owner picks it up. Set `priority:p1` on the database and backend ones since frontend depends on them."*

Expected output: 4–6 child issues, each ending up with one `squad:{member}` label. Now every team member knows what they own.

### Step 6 — Launch Ralph

In a **second terminal** (leave Copilot CLI running in the first):

> **Spoken:** *"Ralph is the polling loop. He wakes up team members and gets them to work the issues — always opening a PR, never committing to main."*

```bash
squad triage --execute --interval 2 \
  --copilot-flags "--yolo --agent squad" \
  --log-file ./ralph.log
```

What Ralph does each round:
1. Pulls the list of open issues.
2. Builds a context snapshot (issues + squad state + recent decisions) and writes it to a temp file.
3. Invokes `gh copilot -p <context.md> --agent squad`.
4. The Squad coordinator inside that invocation picks the next-best issue, reads `.squad/routing.md`, identifies the `squad:{member}`, and **spawns that specialist** to do the work.
5. The specialist creates branch `squad/<issue-number>-<slug>`, commits, opens a PR with `Closes #<n>`, links the PR back on the issue.
6. Ralph monitors PR state (`needsReview` → `readyToMerge` → `done`), retries with 4-tier escalation on failure, and pauses for human review when needed.

Show `--health` in another window for visual oomph:
```bash
squad triage --health
```

### Step 7 — Review and merge

PRs start landing. For each:
- Show the diff to the audience.
- If you've set branch protection, click "Approve" + "Merge" yourself. Otherwise Ralph (or `@copilot`) auto-merges when CI passes.
- Once all PRs land, refresh `/favorites` in the running app to show it working end-to-end.

## 5. Where the work *actually* runs

This is the question audiences ask, and it's worth answering crisply on stage.

```
Your laptop                                    GitHub
─────────────                                  ──────
copilot --agent squad ──┐                      Issues  ◄──────┐
   (you talk here)      │                      Labels         │
                        │                      PRs    ◄────┐  │
squad triage --execute ─┤  reads issues, ──────►          │  │
   (Ralph loop)         │  spawns specialists  Cloud      │  │
                        │     │                @copilot ──┘  │
                        ▼     ▼                  (optional)  │
                   ┌─────────────────┐                       │
                   │ Specialist agent│ ─────► branch ────────┘
                   │ (runs locally,  │        + PR
                   │  acts as Edie,  │
                   │  McManus, …)    │
                   └─────────────────┘
                            ▲
                       Scribe writes
                       back to .squad/
```

- **Default:** specialists run **locally** under Ralph's control (you see the terminal stream — great for stage demos).
- **Optional acceleration:** `squad copilot` registers `@copilot` as the coding agent. For 🟢 "good fit" issues (bug fixes, small features, boilerplate, doc updates) Squad can hand off to the **GitHub Copilot cloud coding agent**, which runs in a sandbox on github.com and opens PRs in parallel. The team's `team.md` defines what's 🟢 / 🟡 / 🔴.
- **Hybrid (recommended for live demos):** keep Ralph local so the audience watches things happen, but let him farm out clearly-isolated 🟢 issues to `@copilot` so multiple PRs flow in parallel without your laptop being the bottleneck.

The framing on stage: *"Ralph runs locally, but he can delegate the actual coding to the cloud agent for clearly-scoped tickets, so the team works in parallel."*

## 6. PR-first guardrails

Two layers, use both:

1. **Built into Squad.** `.squad/routing.md` and `issue-lifecycle.md` already encode the `squad/{issue}-{slug}` branch + PR + `Closes #` convention. The Squad agent obeys it and the `copilot-instructions.md` Squad ships reinforces it for `@copilot`.
2. **GitHub-level (optional).** Branch protection on `main`:
   ```bash
   gh api -X PUT "repos/:owner/:repo/branches/main/protection" \
     -F required_pull_request_reviews.required_approving_review_count=1 \
     -F enforce_admins=false \
     -F required_status_checks= \
     -F restrictions=
   ```
   Or via the UI: Settings → Branches → Add rule for `main` → "Require a pull request before merging".

If you set branch protection, **you** become the reviewer for incoming PRs — factor in time to approve on stage.

## 7. Reset script — re-running the demo cleanly

After a demo you'll want to wipe state and start over.

```powershell
# From inside squad-webshop/

# 1. Stop Ralph if it is running
Get-Process | Where-Object { $_.CommandLine -match 'squad triage' } | Stop-Process

# 2. Delete the GitHub repo (irreversible — confirm first!)
$REPO = (gh repo view --json nameWithOwner -q .nameWithOwner)
Write-Host "About to delete: $REPO"
gh repo delete $REPO --yes

# 3. Remove the remote and reset to a clean local state
git remote remove origin

# 4. Reset to the initial commit (drop demo commits)
$INITIAL = (git rev-list --max-parents=0 HEAD | Select-Object -First 1)
git reset --hard $INITIAL

# 5. Delete demo branches
git branch | Where-Object { $_ -notmatch '\*\s*main' } | ForEach-Object {
  git branch -D ($_.Trim())
}

# 6. Wipe the Squad team so the next demo gets a fresh cast
Remove-Item -Recurse -Force .squad -ErrorAction SilentlyContinue
Remove-Item squad.agent.md -ErrorAction SilentlyContinue
Remove-Item ralph.log -ErrorAction SilentlyContinue

# 7. Wipe the local DB so user accounts/orders from the previous demo are gone
Remove-Item server\data\webshop.db -ErrorAction SilentlyContinue
```

Verify: `git log --oneline` shows only the initial commit, `git remote -v` is empty, no `.squad/` folder, no GitHub repo by that name.

## 8. Troubleshooting

| Symptom | Likely cause / fix |
|---|---|
| `gh repo create` fails: "name already exists" | Run §7 reset, or pick a different name. |
| `squad init` complains about an existing team | It's idempotent; safe to run again. To start fresh, `rm -rf .squad squad.agent.md` first. |
| Lead never triages new issues | Make sure they have the `squad` label (not `squad:untriaged`). Check with `gh issue list --label squad`. Or nudge the team in Copilot: *"Lead, there's an untriaged issue."* |
| Ralph never picks up an issue | Confirm the issue carries a `squad:{member}` label (not just `squad`). Triage hasn't happened yet. |
| Ralph picks the same issue every round | He didn't transition state. Check `ralph.log` and the issue's labels — Ralph should remove `squad:{member}` once a PR is open and the state moves to `needsReview`. |
| `@copilot` cloud agent not picking up an issue | Issue must be **assigned** to `@copilot` (not just labeled). Run `squad copilot --auto-assign` once. |
| `--yolo` still asks for confirmation | Some destructive ops are always gated. Approve and move on. |
| PRs keep failing CI | Show the audience how Squad handles it: comment *"Tester — the PR for #12 is failing, please debug"* in Copilot. The specialist iterates. |
| App doesn't start after the demo | `npm install` again (the agent may have added deps), then `npm run dev`. |

## 9. What "good" looks like at the end

- 1 closed PRD issue tagged `squad` (then closed)
- 4–6 closed work-item issues, each had a `squad:{member}` label and was closed by its PR
- 4–6 merged PRs on `main`, each authored by a named specialist (or `@copilot`)
- A working `/favorites` page in the running app
- A `favorites` table in `server/data/webshop.db`
- Updated README mentioning the favorites feature
- `.squad/decisions.md` with new entries the Scribe wrote during the demo

If all of those exist and the app still runs, the demo worked.
